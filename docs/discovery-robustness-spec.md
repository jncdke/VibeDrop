# 局域网发现鲁棒性增强规格

## 1. 背景

当前手机端自动发现桌面端的实现是：

1. 获取手机当前 IPv4
2. 假设局域网是当前地址所在的 `/24`
3. 对 `x.x.x.1` 到 `x.x.x.254` 逐个请求 `http://IP:9001/discover`
4. 谁返回合法 JSON，就视为发现到桌面端

这套方案实现简单，但存在天然问题：

- 依赖 `/24` 假设，不适合更复杂的网段
- 发现是单次轮询，不是持续服务公告
- 每个地址都要建立 HTTP 请求，容易受瞬时网络波动影响
- 没有“主动应答”机制，发现成功率受超时参数影响很大

用户最终要的是：

- 新手机安装 App 后，尽量直接发现桌面端并发起配对
- 同一局域网下，发现过程更稳，不要因为一次扫描抖动就漏掉
- 即使广播不可用，也保留兜底发现能力

## 2. 目标

本次增强目标：

1. 手机端优先使用 UDP 广播主动探测附近桌面端
2. 桌面端监听 UDP 发现请求并立即回包
3. 手机端保留现有 HTTP `/discover` 扫描作为兜底
4. 两种来源的结果合并去重，继续沿用现有前端配对流程

本次明确不做：

- 桌面端主动扫描手机端
- mDNS / Bonjour / NSD 全平台服务公告
- 跨三层网络或跨子网发现
- 浏览器模式下的自动发现

## 3. 设计原则

### 3.1 发现机制要“主动探测 + 被动兜底”

更稳的做法不是只依赖单次扫段，而是分成两层：

- 第一层：UDP 广播探测，桌面端立即应答
- 第二层：HTTP `/discover` 轮询兜底

这样可以兼顾：

- 广播可用时的快速、低成本发现
- 广播被路由器或系统限制时的兼容性

### 3.2 不推翻现有前端发现模型

前端仍然使用 `discoverNearbyDesktops()` 和现有配对/保存/连接流程。

变化只应发生在原生命令 `discover_desktops` 的内部实现：

- 返回结构保持兼容
- 前端不需要重写发现 UI

### 3.3 保持协议极简

UDP 发现协议只解决“谁在局域网里运行桌面端”这个问题，不掺杂 PIN、配对状态或客户端身份校验。

返回内容继续围绕：

- `kind`
- `server_id`
- `hostname`
- `ip`
- `port`
- `protocol_version`

## 4. 协议设计

### 4.1 端口

- 继续使用桌面端现有服务端口 `9001`
- TCP `9001` 继续提供 HTTP/WebSocket
- UDP `9001` 新增发现应答

同一端口号但不同传输层协议不会冲突。

### 4.2 手机 -> 桌面 UDP 探测报文

JSON：

```json
{
  "kind": "discover_probe",
  "protocol_version": 1
}
```

### 4.3 桌面 -> 手机 UDP 响应报文

JSON：

```json
{
  "kind": "desktop",
  "server_id": "desktop-xxxx",
  "hostname": "MacBook-Air.local",
  "ip": "192.168.31.45",
  "port": 9001,
  "protocol_version": 1
}
```

## 5. 实施方案

### 5.1 桌面端

桌面端新增一个 UDP 监听任务：

1. 绑定 `0.0.0.0:9001/udp`
2. 循环接收数据报
3. 解析为 JSON
4. 若 `kind == discover_probe` 且协议版本兼容，则向来源地址回发桌面信息

要求：

- 不影响现有 HTTP / WebSocket 服务
- 解析失败时忽略，不中断监听循环
- 监听失败时写日志，但不影响桌面端主服务启动

### 5.2 手机端

`discover_desktops` 改成混合发现，并在二阶段升级为“多候选网卡发现”：

1. 枚举系统可见的 AF_INET 网络接口地址
2. 过滤掉不适合局域网发现的 IPv4：
   - `loopback`
   - `link-local`
   - 未指定地址
3. 保留所有可疑似局域网的候选 IPv4，而不是只赌一张 `local_ip()`
4. 创建 UDP socket，开启广播
5. 向以下目标各发送多次探测包：
   - `255.255.255.255:9001`
   - 每个候选 IPv4 所在 `/24` 推导出的 `x.x.x.255:9001`
6. 在一个短时间窗口内接收 UDP 响应并解析结果
7. 再对所有候选 IPv4 所在的 `/24` 执行 HTTP `/discover` 兜底扫描
8. 合并两种结果并按 `server_id` 去重

### 5.3 去重与排序

合并策略：

- 以 `server_id` 为主键
- 若 `server_id` 为空，则退化到 `ip:port`
- 优先保留 UDP 发现结果中的实时 IP

排序继续按：

- `hostname`
- `ip`

### 5.4 超时策略

建议参数：

- UDP 广播发送间隔：极短
- UDP 接收窗口：约 900ms
- HTTP 单地址超时：维持现有量级
- HTTP 作为兜底，不取消

设计意图：

- 让局域网里正常运行的桌面端通常先通过 UDP 被发现
- 让广播受限环境仍然能靠 HTTP 扫描找到

## 6. 行为变化

改造后实际用户感受应该是：

1. 新手机安装 App 后点击扫描，更快看到桌面端
2. 即使某一轮广播没收到，也还有 HTTP 兜底
3. Android 真机即使存在多张网卡地址，也不再因为拿错一张 IP 就整轮扫偏
4. 已保存的设备逻辑、配对逻辑、连接逻辑都不需要改动

## 7. 诊断日志

为了让真机排障不再靠猜，发现链路需要补运行日志，统一前缀：

- `[VibeDrop discover]`

至少输出：

1. 本轮识别到的候选接口与 IPv4
2. UDP 广播目标数量
3. HTTP 扫描目标数量
4. UDP / HTTP / merged 三段结果数量

这样当用户反馈“同 Wi-Fi 还是扫不到”时，可以快速区分是：

- 拿错本地网卡
- 广播没收到应答
- HTTP 扫描没命中
- 前端展示层状态问题

## 8. 前端原生能力探测

这次真机排查暴露出另一个更上游的问题：即使原生发现实现已经可用，如果前端把当前环境误判成“浏览器模式”，扫描入口也不会调用原生命令。

因此前端不能只用下面这一种形态判断原生能力：

- `window.__TAURI__.core.invoke`

需要升级为“统一原生调用入口”，兼容：

1. `window.__TAURI__.core.invoke`
2. `window.__TAURI_INTERNALS__.invoke`

设计要求：

- `supportsNativeFileReceive()` 改为基于统一调用入口判断
- `invokeNative(...)` 只通过统一入口发命令
- 持久化历史、附近发现、分块落盘等所有原生命令统一复用这条入口
- 如果 bridge 是页面启动后迟到补上的，前端需要主动刷新与桌面端的连接能力上报，避免桌面端继续停留在旧的 `can_receive_files=false` 认证快照上

这样才能避免“App 已安装，但前端自认为是浏览器模式”的错判。

## 9. Android WebView 宿主兜底

这次真机进一步确认了另一个底层风险：在部分 Android WebView / brownfield 组合下，Tauri 的 init scripts 可能没有按预期注入到页面，表现为：

- `window.ipc.postMessage` 存在
- `window.__TAURI_INTERNALS__` 只剩插件占位或部分字段
- `window.__TAURI_INTERNALS__.invoke` 与 `window.__TAURI__` 均缺失

这说明不是“没有底层 IPC 通道”，而是“高层 JS bridge 初始化没跑完”。

因此 Android 宿主层需要补一个兜底机制：

1. 页面 `onPageFinished` 后检查 bridge 状态
2. 如果缺少 `window.__TAURI_INTERNALS__.invoke`，则手动执行 `RustWebView.initScripts`
3. 只补跑一次，避免重复污染全局
4. 补跑后再次记录 bridge 状态，确认 `invoke` 已恢复

这个机制的本质是：不重写 Tauri 协议，不自造 IPC，只在宿主层确保 Tauri 自己生成的初始化脚本真正进入页面。

## 10. 边界

本次增强不能保证下面这些场景：

- 路由器开启 AP 隔离 / 客户端隔离
- 手机和电脑虽然同 SSID，但被放到不同 VLAN / 子网
- 桌面端本身没有运行或被防火墙阻断 `9001`
- 浏览器模式使用网页而不是安装后的 App
- 桌面端主动扫描附近手机并直接列出新手机

## 11. 后续演进

如果后续还要再进化，可继续做：

1. mDNS / Bonjour 服务公告
2. 桌面端显示“最近发现到的手机”
3. 多网段掩码感知，而不是固定 `/24`
4. 后台定时发现与连接恢复
