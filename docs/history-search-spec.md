# 历史记录搜索功能规格

## 1. 背景与目标

当前项目的历史页已经具备以下能力：

- 设备筛选
- 快速时间筛选
- 高级时间/类型/状态筛选
- 热力图小时筛选

但缺少“按关键词直接定位历史记录”的能力。用户只能先靠时间或类型缩小范围，再人工逐条查找，效率较低。

本次新增“历史记录搜索功能”的目标是：

1. 让用户可以直接按关键词检索历史记录。
2. 搜索结果与现有筛选、热力图、空态、统计摘要保持一致。
3. 移动端历史页与桌面端历史页保持一致的核心行为，避免体验割裂。
4. 优先复用现有前端本地过滤链路，不引入新的存储结构或后端接口。

## 2. 范围

本次范围内：

- Android / 移动端历史页 `mobile/src/*`
- 浏览器镜像历史页 `desktop/static/*`
- Mac 桌面端历史页 `desktop/src/*`

本次范围外：

- 后端 Rust 存储结构变更
- 服务端搜索接口
- 搜索高亮
- 搜索历史、联想词、拼音检索、正则检索

## 3. 用户场景

### 场景 A：按文字内容查找

用户想找曾经发送过的一段文本，例如 “会议纪要”。

期望：

- 输入关键词后，历史列表立即只显示匹配的记录。
- 热力图和筛选摘要也同步反映搜索后的结果集。

### 场景 B：按设备名查找

用户记得内容发给了某台电脑，但不记得时间，例如 “Mac mini”。

期望：

- 搜索框支持匹配目标设备名 / 来源设备名。

### 场景 C：按文件名查找

用户想找曾经传过的 `demo.mp4` 或 `合同.pdf`。

期望：

- 搜索框可匹配记录主文本和媒体/文件项文件名。

## 4. 交互设计

### 4.1 搜索框位置

- 移动端：放在历史页标题下方、设备筛选按钮上方。
- 桌面端：放在历史页标题区下方、设备筛选条上方。

这样做的原因：

- 搜索属于“一级入口”，使用频率高，应该比高级筛选更直接。
- 搜索和筛选都属于“缩小结果集”，位置相邻更符合用户心智。

### 4.2 输入行为

- 输入框 placeholder 使用“搜索内容、设备或文件名”。
- 输入时实时生效，不需要点击“搜索”按钮。
- 去掉首尾空格后为空时，视为未启用搜索。
- 提供清空按钮，一键清空当前搜索词。

### 4.3 搜索与现有筛选的关系

过滤顺序统一为：

1. 设备筛选
2. 快速时间筛选
3. 自定义日期/时段筛选
4. 类型/状态筛选
5. 关键词搜索
6. 热力图小时选择

原因：

- 搜索属于“基础结果集过滤”的一部分，应先得到基础结果，再做热力图小时级二次筛选。
- 这样热力图展示的是“搜索后的活动分布”，不会出现图和列表不一致。

### 4.4 摘要与空态

- 当搜索词非空时，筛选摘要需要新增一个“搜索：xxx”标签。
- 清除全部筛选时，也应清空搜索词。
- 当仅因搜索无结果时，空态文案显示“没有匹配的历史记录”。
- 若同时存在热力图小时筛选且无结果，优先保留现有“这个时段没有符合条件的记录”逻辑。

## 5. 匹配规则

### 5.1 标准化规则

搜索前对输入和记录字段做统一标准化：

- 转字符串
- `trim()`
- 转小写
- 合并连续空白

### 5.2 分词规则

- 以空白字符分词
- 多个关键词采用 AND 语义

例子：

- 搜索 `mac demo` 时，要求同一条记录的可搜索文本中同时包含 `mac` 和 `demo`

这样做比 OR 更符合“缩小结果集”的直觉，也更适合历史定位。

### 5.3 参与匹配的字段

移动端历史记录：

- `entry.text`
- `entry.targetAlias`
- `entry.targetName`
- `entry.targetDeviceName`
- `entry.fileName`
- `entry.items[].fileName`
- 记录类型中文标签
- 记录状态中文标签

桌面端历史记录：

- `entry.text`
- `entry.client_name`
- `entry.client_ip`
- `entry.file_name`
- `entry.items[].file_name`
- 记录类型中文标签

说明：

- 把“类型/状态标签”也纳入搜索，可兼容用户输入“图片”“失败”等自然语言检索。
- 不单独搜索时间字符串，时间已有专门筛选入口，避免匹配噪音。

## 6. 技术方案

### 6.1 为什么只改前端

当前历史数据在历史页渲染前已经全部加载到前端内存：

- 移动端：`getHistory()`
- 桌面端：`getLog() -> normalizeLogEntry()`

因此搜索最合适的实现位置是前端过滤函数：

- 移动端：`filterHistoryEntries(...)`
- 桌面端：`filterLogEntries(...)`

优点：

- 不改存储格式
- 不改 Rust IPC
- 搜索延迟低
- 和现有筛选链路天然复用

### 6.2 代码结构方案

新增统一的搜索辅助函数：

- `normalizeSearchText(value)`
- `tokenizeSearchQuery(query)`
- `buildHistorySearchIndex(entry)` / `buildLogSearchIndex(entry)`
- `matchesHistorySearch(entry, filters)`
- `matchesLogSearch(entry, filters)`

原则：

- 搜索字段拼接逻辑集中，避免散落在渲染代码中。
- 过滤函数只负责任务编排，不直接堆砌字段判断。

### 6.3 状态扩展

在过滤状态对象中新增：

- 移动端 `currentHistoryFilters.query`
- 桌面端 `currentLogHistoryFilters.query`

默认值均为空字符串。

### 6.4 UI 方案

搜索框采用现有玻璃拟态风格，新增：

- 输入框
- 搜索图标文案区
- 清空按钮

移动端和桌面端视觉风格保持各自页面一致，但交互规则一致。

## 7. 兼容性与风险

### 7.1 兼容性

- 老历史数据没有新增字段也不受影响，因为搜索索引会对缺失字段做空串兜底。
- 导入/导出格式无需修改。

### 7.2 风险

风险 1：搜索与热力图统计不一致

- 规避：把搜索放进基础过滤链路，在热力图渲染前完成。

风险 2：移动端与 `desktop/static` 镜像逻辑漂移

- 规避：同一轮修改同步更新两套前端文件。

风险 3：输入过程中频繁重绘

- 规避：继续复用现有 `scheduleHistoryRender()` / `renderLog()`，不引入额外异步状态。

## 8. 实施步骤

1. 为移动端和桌面端过滤状态增加 `query` 字段。
2. 为移动端历史页、桌面端历史页补搜索框 UI。
3. 新增搜索辅助函数和匹配逻辑。
4. 将搜索匹配接入现有 `filterHistoryEntries` / `filterLogEntries`。
5. 更新筛选摘要、空态、清空逻辑。
6. 做语法检查与交互回归检查。

## 9. 验收标准

满足以下条件即视为完成：

1. 移动端和桌面端历史页都能按关键词实时筛选。
2. 能匹配文本内容、设备名、文件名。
3. 搜索可与时间/类型/状态/设备筛选叠加使用。
4. 热力图统计和列表结果基于同一搜索后结果集。
5. “清除筛选”会同时清掉搜索词。
6. 无搜索结果时显示明确空态。
