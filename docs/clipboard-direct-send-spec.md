# 剪贴板直发优化规格

## 背景

当前发送文字到 Mac 的流程要求用户先把手机剪贴板内容粘贴到设备卡片的输入框，再点击“发送”或“发送并回车”。高频场景下，用户通常已经在手机系统剪贴板中准备好了文本，因此这一步粘贴可以合并到发送按钮动作中。

## 目标

在输入框为空时，点击“发送”或“发送并回车”自动读取手机剪贴板文本并发送到目标 Mac。输入框已有文本时，仍以输入框文本为准，避免按钮点击意外覆盖用户已经编辑好的内容。

## 行为规格

1. “发送”按钮：
   - 输入框非空：发送输入框文本，保持现有行为。
   - 输入框为空：读取手机剪贴板文本，剪贴板非空则发送该文本。
   - 输入框和剪贴板都为空，或剪贴板读取失败：不发送，并提示用户。

2. “发送并回车”按钮：
   - 文本来源规则与“发送”一致。
   - 发送成功后继续触发 Mac 端回车，沿用现有 `type_enter` 协议。

3. “回车”按钮：
   - 不读取剪贴板，只发送回车，保持现有行为。

4. 发送历史：
   - 发送剪贴板文本时，也记录到历史记录，文本内容为实际发送文本。

5. 输入框清理：
   - 发送成功后沿用现有清空输入框逻辑。
   - 剪贴板直发时输入框本来为空，不需要额外写入再清空。

## 技术方案

前端新增 `resolveTextToSend(deviceId)`，负责按“输入框优先，剪贴板兜底”的顺序返回文本。剪贴板读取新增 `readClipboardText()`，优先使用 Android WebView 注入的 `window.NativeClipboard.readText()`，再兜底到 Tauri `clipboardManager.readText()` 和浏览器 `navigator.clipboard.readText()`。

Android 原生层在现有 `NativeClipboard` JS Bridge 中补充 `readText()`，通过 `ClipboardManager.primaryClip` 获取第一项并用 `coerceToText(context)` 转为字符串。该能力只在用户点击发送按钮后、App 前台时触发，减少 Android 剪贴板隐私提示和权限限制带来的干扰。

## 非目标

- 不改变 Mac 端 WebSocket 协议。
- 不自动轮询手机剪贴板。
- 不在 UI 上新增“粘贴”按钮。
- 不改变“回车”按钮的单独行为。
