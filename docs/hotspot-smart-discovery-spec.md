# 热点智能发现匹配规格

## 背景

手机开热点、Mac 连接热点后，Mac 的局域网地址会从原来的普通 Wi-Fi 地址切换到热点网段地址。当前传输协议本身支持这种拓扑，但已保存设备仍可能保留旧地址，例如 `192.168.31.45`，导致 App 持续重连旧地址而失败。

真机排查还发现了一个更隐蔽的问题：Android 前台页面里的保存设备列表和原生后台剪贴板服务使用的是两份状态。启动时如果先把“旧保存地址”同步给原生后台，再由智能发现慢慢把前台地址修正过来，就会出现前台已经连到新地址、后台服务却还在死连旧地址的窗口期。

## 目标

1. App 启动连接已保存设备前，先静默发现附近桌面端，并用发现结果刷新已保存设备的 IP。
2. WebSocket 连接失败或断开重连时，自动触发带冷却的静默发现，避免热点/Wi-Fi 切换后一直死连旧地址。
3. 发现到的桌面端优先按 `server_id` 匹配已保存设备；缺少 `server_id` 时，用 hostname、历史 hostname 和弱名称匹配兜底。
4. 匹配成功后自动更新 `ip`、`port`、`hostName`、`serverId`，保存配置，同步 Android 后台剪贴板配置，并重连当前设备。
5. Android 原生后台剪贴板配置必须以后台发现刷新后的设备列表为准，不能先于智能发现同步旧 endpoint。

## 非目标

1. 不引入云中转或互联网发现。
2. 不改变现有 WebSocket/PIN/配对协议。
3. 不保证手机热点禁用客户端互访、系统防火墙阻断或 VPN 强制拦截本地网络时可用。

## 实施方案

### 启动连接前刷新

`loadSettings()` 渲染设备后不直接用旧 IP 连接，而是调用带“发现后再同步后台配置”的 `connectAll()`。如果原生发现可用，它先执行 `discover_desktops`，再把发现刷新后的设备列表同时喂给：

1. 前台 WebSocket 主连接。
2. Android 原生后台剪贴板配置。

如果原生能力不可用，则保持旧的直接连接行为。

### 后台配置以发现结果为准

新增一条明确的同步语义：

1. 当准备执行启动发现刷新时，先清空原生后台剪贴板目标，避免后台服务继续对过期地址死连。
2. 智能发现完成后，再把当前最新保存设备列表同步给原生后台服务。
3. 如果这次发现没有刷新出新 IP，也要在发现结束后重新同步一次当前设备列表，确保原生后台和前台设置页不再长期分叉。

这样可以把“旧配置先同步给原生后台、发现完成后才改前台”的反向顺序改成“发现结果成为单一真源，前后台同时切换”。

### 原生冷启动门控

为了解决 `KeepAliveService` 早于 WebView 启动的问题，原生层增加一个“前端同步门控”：

1. `MainActivity` 首次冷启动时，先写入一个“等待前端同步”的门控标记，再启动 `KeepAliveService`。
2. `KeepAliveService` / `BackgroundClipboardSyncManager` 在门控有效期内不读取持久化后台配置，也不主动连接旧 endpoint。
3. 前端第一次调用 `NativeBackgroundClipboard.syncConfig()` 时，无论同步的是空设备列表还是发现后的最新地址，都先解除门控，再按当前配置刷新后台连接。
4. 如果前端异常、WebView 卡死或发现流程未按时完成，则在超时后自动放行，回退到“使用持久化配置启动后台连接”的旧行为，避免后台能力被永久卡住。

推荐超时：

1. 门控超时设为 20 秒，覆盖冷启动 + WebView 初始化 + 智能发现的常见耗时。
2. 门控只在当前 App 进程的首次前台冷启动时生效，不在同一进程内的普通返回前台、Activity 重建、服务刷新时重复打断现有后台连接。

### 发现范围收敛

当前 Android 端的原生发现会从所有私网 IPv4 接口推导扫描目标，这在手机开启 VPN 时会把 `tun0 172.19.0.1/30` 之类的点对点接口也当成局域网来做 `/24` HTTP 扫描，导致大量无效 `:9001` 探测。

新的收敛策略：

1. 候选接口按名称做分层，优先保留 `wlan*`、`ap*`、`swlan*`、`softap*`、`eth*`、`en*`、`bridge*`、`br*`、`rndis*` 这类明显承载局域网的接口。
2. `tun*`、`ppp*`、`rmnet*`、`ccmni*`、`clat*`、`dummy*`、`ifb*` 等明显属于 VPN、蜂窝、虚拟或点对点链路的接口，不参与 HTTP 子网扫描，也不生成定向广播地址。
3. UDP 发现保留全局广播 `255.255.255.255:9001`，并仅为“可局域网广播”的优先接口追加定向广播地址。
4. HTTP 子网扫描只对优先接口生成目标，并对重复子网去重；默认最多扫描 2 个优先子网。
5. 如果一个优先接口都识别不出来，再回退到现有“使用所有候选私网 IP”逻辑，避免因为 OEM 特殊接口命名而完全失去发现能力。

设计取舍：

1. 这里优先解决“错误扫描过宽”而不是一开始就做完整网掩码感知，因为当前根因主要来自 VPN `tun0` 被误当局域网。
2. `/24` 枚举逻辑暂时保留在“已选中的真实局域网接口”上，这样改动小、风险可控，同时能显著减少无效探测。

### 分层发现

当前即使只扫描真实局域网接口，冷启动时仍会对整个 `/24` 执行 HTTP `/discover` 探测，真机上表现为一次启动约 253 个 HTTP 目标。这在“已保存设备地址本来就没变”的大多数启动场景下属于不必要的开销。

新的分层策略：

1. 手动“扫描附近电脑”继续保留完整发现，目标是列出附近所有桌面端，因此仍允许执行 UDP 广播 + 局域网 HTTP 扫描。
2. 启动连接前、后台配置对齐、断线重连前的“智能刷新已保存设备”走节省流量的优先路径：先广播 UDP，再优先探测已保存设备的直连线索。
3. 已保存设备的直连线索至少包含：
   1. 当前保存的 `ip:port`。
   2. 已知 `hostName` / 历史 hostname / 当前连接回传 hostname。
   3. `serverId` 仅用于校验响应是否还是同一台机器，不直接参与网络寻址。
4. 原生层对每个已保存设备先做定向 HTTP `/discover` 探测：
   1. 如果探测目标是 IP，则用该 IP 直接请求。
   2. 如果探测目标是 hostname，例如 `*.local`，则直接以 hostname 作为 URL host 交给系统解析。
5. 只有当“至少一个已保存设备的直连线索未被验证成功”时，才回退到局域网 `/24` HTTP 扫描。
6. 如果所有具备直连线索的已保存设备都已经验证成功，则跳过局域网 `/24` 扫描，直接把这些发现结果交给现有匹配逻辑更新配置。

验证判定：

1. “验证成功”不等于“这个目标返回了一个桌面端”这么粗糙，而是要与已保存设备身份做一致性校验。
2. 优先用 `serverId` 做强校验；如果已保存设备没有 `serverId`，则退化为 hostname 一致或目标 IP 一致。
3. 如果旧 IP 已经被别的机器占用，但返回的 `serverId` / hostname 与已保存设备不一致，则该定向探测必须判定为失败，并继续进入子网扫描兜底。

设计取舍：

1. 这不是彻底替代完整扫描，而是把“昂贵的完整扫描”从默认路径降级为兜底路径。
2. 这样可以保留热点切网后的自愈能力，因为旧 IP 失效或身份校验失败时，仍然会继续扫真实局域网子网。
3. 常态启动下，如果 `.45`、`.15` 这类已保存 endpoint 仍然有效，就不再需要每次都额外扫 253 个 HTTP 目标。

### 应用内诊断面板

前面的修复已经把“配置同步顺序”“原生冷启动门控”“发现范围收敛”“分层发现”都做出来了，但它们目前主要依赖 `adb logcat` 才能看清。这对开发排障有效，对日常使用和后续回归都不够友好。

新增一个仅放在设置页的“连接诊断”面板，目标不是替代日志，而是把最关键的运行态浓缩成可读摘要：

1. 前端层：
   1. 当前保存设备列表。
   2. 每个设备的已保存 `ip:port`、`hostName`、`serverId`。
   3. 当前前台 WebSocket 是否已认证、最近认证 hostname。
2. 原生后台剪贴板层：
   1. 冷启动门控是否仍在生效、剩余时间。
   2. 当前后台配置里有哪些设备。
   3. 每个后台连接当前状态，例如 `idle / connecting / open / authenticated / failed / scheduled_reconnect`。
   4. 当前连接目标 `ip:port`、最近错误、最后状态更新时间。
3. 智能发现层：
   1. 最近一次发现开始时间、结束时间。
   2. 候选接口列表及其 `priority / directed_broadcast / http_sweep`。
   3. 最近一次发现是否走了“定向探测命中并跳过子网扫描”。
   4. `direct targets`、`direct probe verified x/y`、`full_http_targets`、`merged_count` 等摘要数字。
   5. 同时展示“定向目标”和“发现结果”，避免把探测目标误认为最终识别出的桌面端地址。

设计约束：

1. 面板展示“最近一次快照”，不要求成为全量事件时间线。
2. 快照来源必须清晰：
   1. 前端层直接读 JS 当前状态。
   2. 原生后台层通过 `JavascriptInterface` 实时返回。
   3. 智能发现层通过 Tauri 命令返回最近一次 discovery 的原生快照。
3. 面板默认折叠，避免干扰普通用户；点开后才轮询刷新。
4. 轮询频率控制在 1 秒左右，只在设置页且面板展开时刷新，避免无意义开销。

实现分工：

1. Android `BackgroundClipboardSyncManager` 内部维护可序列化的运行态快照。
2. `MainActivity.BackgroundClipboardBridge` 暴露 `getStatus()` 给 WebView。
3. Rust discovery 模块维护最近一次发现的摘要快照，并新增 `get_discovery_diagnostics` 命令。
4. 前端设置页新增“连接诊断”折叠卡片，把三层状态合并渲染成摘要区块。

展示原则：

1. 先给结论，再给细项。例如先显示“后台已认证 2/2”“最近发现走了定向探测并跳过子网扫描”，再列设备和接口明细。
2. 出现错误时明确标记来源层级，例如“后台连接失败”与“前台未认证”不能混成一句泛化提示。
3. 所有时间都显示相对时间和绝对时间里的一个，避免只看到原始时间戳。

### 连接失败触发刷新

`scheduleReconnect()` 不再盲目沿用传入的旧 IP。它会：

1. 请求一次带冷却的智能发现刷新。
2. 到真正重连时重新读取最新保存设备。
3. 如果发现链路已更新 IP，则使用新 IP 重连；否则继续用旧 IP 按原逻辑重试。

### 同步语义

`syncKnownDevicesWithDiscovery()` 继续负责把发现结果写回保存设备。更新后调用 `saveDevices()`，由现有链路同步 Android 后台剪贴板配置；如果该设备当前有连接对象，则立即用新 endpoint 调 `connectDevice()`。

额外约束：

1. 启动阶段不能在发现前立刻调用后台配置同步。
2. 原生 bridge 延迟就绪时，`handleNativeCapabilityReady()` 也必须走同样的“先发现、后同步、再重连”顺序。
3. 原生后台门控解除前，`reloadConfig()` 不能主动读取旧持久化配置。
4. 门控超时放行后，如果前端随后仍然完成同步，后来的同步结果必须覆盖超时放行后的连接目标。
5. 发现候选接口排序必须稳定，避免不同启动轮次因为接口枚举顺序漂移导致扫描范围不一致。
6. 当存在 Wi‑Fi / 热点类接口时，VPN/点对点接口不能参与 HTTP `/24` 枚举。

## 验证

1. 手机普通 Wi-Fi 下配对 Mac。
2. 手机开启热点，Mac 连接手机热点。
3. 确认 Mac 热点地址变为新网段，例如 `192.168.253.251`。
4. 打开 App 后应自动把同一台 Mac 的保存 IP 从旧网段更新到新网段，并连接成功。
5. 断开热点再回到普通 Wi-Fi 后，同一机制应把 IP 刷回普通 Wi-Fi 网段。
6. 冷启动 App 后，原生后台剪贴板日志不应先读取旧持久化配置并建立连接；应先进入等待前端同步或等待超时的状态。
7. 当前端第一次同步空设备列表时，后台应清空连接但不使用旧配置抢跑。
8. 当前端发现完成并同步最新地址后，后台应直接按最新地址建立连接并认证成功。
9. 如果模拟前端未完成同步，门控超时后后台应恢复到旧的持久化配置加载逻辑。
10. 发现未改动设备列表时，前台和原生后台也应保持同一份 endpoint。
11. 在当前真机的 `wlan0=192.168.31.70/24`、`tun0=172.19.0.1/30` 场景下，发现过程不应再对 `172.19.0.*:9001` 做整段 HTTP 扫描。
12. 冷启动采样里，稳定连接应最终收敛到真实桌面端地址，例如 `192.168.31.45:9001`、`192.168.31.15:9001`，而不是出现新的 `ESTAB -> 192.168.31.180:9001`。
13. 在“已保存 IP 仍然正确”的冷启动场景下，发现日志应优先出现“定向探测命中并跳过 HTTP 子网扫描”，而不是每次都出现约 253 个 HTTP 目标。
14. 在“已保存 IP 失效但 hostname / serverId 仍可识别”或“已保存 IP 全部失效”的场景下，发现日志应回退到局域网 HTTP 子网扫描，并最终把设备 IP 修正到最新地址。
15. 设置页展开“连接诊断”后，应能同时看到前端连接状态、原生后台状态和最近一次发现摘要；收起或离开设置页后应停止轮询。
16. 在冷启动门控生效期间，诊断面板应显示门控仍在等待前端同步；门控解除后状态应立即切换。
17. 当最近一次发现走了“定向探测并跳过 HTTP 子网扫描”时，诊断面板应明确展示 `full_http_targets = 0` 或等价表述。
