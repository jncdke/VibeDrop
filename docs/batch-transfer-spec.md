# 批量媒体/文件发送规格

## 1. 背景

当前手机端发送模型是“单项发送”：

- 本地文件选择只取 `FileList` 的第 1 项
- 系统分享只保留 1 个 `pendingSharedContent`
- Android 原生桥只处理 `Intent.ACTION_SEND`

这导致用户无法像成熟传输类产品那样：

- 在相册里一次选中多张照片
- 一次选中多个视频
- 一次批量把多项内容发到 Mac

对 VibeDrop 这类“手机 -> 电脑传输”产品，这属于高频核心需求，不是边缘功能。

## 2. 目标

第一阶段目标：

1. 支持本地文件选择多项发送
2. 支持 Android 系统分享多项内容到 VibeDrop
3. 多项内容按顺序稳定发送到 `VibeDrop 收件箱`
4. 历史记录以“批次”形式展示，而不是刷出一长串零散项

第一阶段明确不做：

- 多项图片写入 Mac 剪贴板
- 多文件并行上传
- 断点续传到单个批次中的任意项
- 批量发送的自定义排序、移除单项、重排

## 3. 设计原则

### 3.1 批量优先落到收件箱，不走剪贴板

单张图片进 Mac 剪贴板是自然语义。

多张图片或多项媒体进“剪贴板”不自然，因为系统剪贴板本质上通常承载的是一个主对象，而不是一组长期待处理的媒体集合。

因此第一阶段专业做法是：

- 单张图片：保留“传图到剪贴板”
- 多项媒体/文件：统一走“传到收件箱”

### 3.2 批量是调度层能力，不是传输协议重写

当前手机到 Mac 的大文件分块协议已经支持稳定发送单个大文件。

批量发送第一阶段不需要重造协议，而是：

- 一次选择多项
- 建立一个批次会话
- 用现有单文件分块协议逐项顺序发送

这比做多文件并发更稳、更容易保证顺序和错误处理。

### 3.3 历史记录按批次聚合

批量传输如果仍然每项都单独刷一条历史，会显得嘈杂，也不利于用户理解“这是同一次操作”。

因此应当：

- 以 1 条批次历史为主
- 条目内保存 `items[]`
- 每个 item 保存各自状态、文件名、路径和错误

## 4. 第一阶段需求规格

### 4.1 选择入口

1. 发送页“收件箱”入口支持多选文件
2. Android 分享入口支持接收多项内容
3. 多项内容可以混合包含图片、视频和普通文件

### 4.2 发送行为

1. 批量发送按顺序逐项执行
2. 任一项失败，不影响后续项继续发送
3. 最终返回批次结果：
   - 全成功
   - 部分成功
   - 全失败

### 4.3 历史记录

1. 批量发送生成 1 条聚合历史记录
2. 历史记录中保存：
   - `itemCount`
   - `items[]`
   - 每项的 `status / savedPath / error`
3. 批次标题按内容自动概括：
   - `[图片] 8 张`
   - `[视频] 3 个`
   - `[媒体] 12 项`
   - `[文件] 6 项`

### 4.4 交互文案

1. 多项系统分享时显示“已选择 N 项”
2. 如果当前是多项内容，点击“传图到剪贴板”应给出明确提示，引导使用收件箱发送
3. 批量发送完成后 toast 应给出汇总结果

## 5. 实施方案

### 5.1 前端选择层

发送页文件选择入口改为：

- `<input type="file" multiple>`

单张仍然按单文件流程走。

多张则组装为批次数组：

- `[{ fileName, mimeType, sizeBytes, readChunkBase64, thumbnailDataUrl? }]`

### 5.2 Android 分享层

原生桥从“单个 pendingSharedContent”升级为“多个 pendingSharedContents”。

要求：

1. 同时支持 `Intent.ACTION_SEND`
2. 同时支持 `Intent.ACTION_SEND_MULTIPLE`
3. 从 `EXTRA_STREAM` / `ClipData` 提取全部 URI
4. 每个 URI 都拷贝到缓存目录
5. JS 侧读取时按 `itemIndex + offset + length` 分块读取

### 5.3 发送调度层

新增批量发送调度函数：

1. 建立批次历史记录
2. 逐项调用现有单文件分块上传
3. 每项结束后更新对应 item 状态
4. 最终计算批次总状态并弹汇总 toast

### 5.4 UI 语义

发送页按钮文案应与当前收件箱语义一致，避免继续使用“下载”这类旧词。

## 6. 为什么第一阶段不用并发上传

因为并发会立刻引入这些复杂度：

- 单连接 WebSocket 缓冲竞争
- 多文件进度展示冲突
- 失败重试粒度复杂
- 历史与最终顺序更难理解

而顺序发送在真实用户感受上已经足够自然，尤其是在本地局域网环境下。

## 7. 后续演进

第二阶段可继续扩展：

- 批次面板中移除单项后再发
- 批次排序
- 批次暂停 / 继续
- 批次失败项重试
- 只对图片/视频生成批量缩略图
