# 连接稳定性优化规格

## 背景

发送页前台连接、原生后台剪贴板连接、Mac 端 WebSocket 服务共同决定设备是否显示为可发送。当前用户观察到手机端绿灯频繁变灰、自动重连抖动。近期 Android logcat 显示后台剪贴板正在每 3 秒反复连接 `minideMac-mini.local / 192.168.3.2:9001` 并失败；代码审查同时发现前台 JavaScript 心跳存在重叠 timeout 的误断风险。

## 已确认现象

1. 前台发送连接每 5 秒发送一次 `ping`，每次创建一个 10 秒 `timeout`，但旧 timeout 没有在发送下一次 ping 前清理。
2. 如果网络、WebView 调度、Mac 端事件循环稍微延迟，旧 timeout 可能在连接已恢复或后续 pong 已返回后仍然触发，从而主动 `close()` 一个可用 WebSocket。
3. 后台剪贴板连接失败后固定 3 秒重连；当保存设备不可达或端口响应异常时，会持续刷失败日志并反复占用连接资源。
4. Mac 当前 `~/.vibedrop/history.jsonl` 只记录成功收发历史；`~/.vibedrop/debug.log` 主要记录拖拽/文件传输，没有持久记录 WebSocket open/auth/close。

## 目标

1. 降低前台连接被客户端误判超时并主动关闭的概率。
2. 后台剪贴板对持续失败的设备逐步放慢重连，不再每 3 秒无止境猛连。
3. 连接成功后重置失败计数，让恢复仍然足够快。
4. Mac 端开始持久记录 WebSocket 关键生命周期，方便以后定位真实断联原因。
5. 不改变现有协议字段，不影响已安装手机端和桌面端的基础兼容性。

## 非目标

1. 不把 UI 强行保持绿色来掩盖真实断联；发送通道不可用时仍应显示连接中或断开。
2. 不删除用户已保存设备；如果 Mac mini 当前不可达，只做退避和诊断，不替用户移除配置。
3. 不在日志里记录用户发送正文或剪贴板正文，避免敏感内容落盘。

## 技术方案

前台连接改为单待确认心跳：`HEARTBEAT_INTERVAL` 调整为 15 秒，`HEARTBEAT_TIMEOUT` 调整为 45 秒。每条连接只允许一个待确认 ping；任意服务端消息都会刷新 `lastMessageAt`。如果已有 ping 等待 pong，就不再叠加新的 timeout，而是检查等待时间是否超过阈值，超时才关闭 socket。

前台重连改为指数退避：失败后从 2 秒开始，逐步增长到 30 秒；认证成功后清零。这样短抖动能快速恢复，持续不可达时不会制造高频重连。

后台剪贴板连接改为指数退避：失败后 3 秒、6 秒、12 秒、24 秒、最多 60 秒；认证成功后重置失败计数。诊断状态继续保留最近错误和等待重连状态。

前台和后台连接回调都加入当前 socket 校验：当旧连接被主动关闭或被新连接替换后，旧 socket 迟到的 `close/failure/message` 回调不能再修改当前连接状态，也不能额外触发重连。

Mac 端在 `debug.log` 写入 `websocket` scope 的关键事件：`open`、`auth_ok`、`auth_failed`、`close`。日志记录 session id、客户端名、角色、认证状态等元数据，不记录消息正文。

## 验收标准

1. `mobile/src/app.js` 和 `desktop/static/app.js` 通过 `node --check`。
2. Android Kotlin 代码能通过 Gradle 编译检查或部署脚本编译。
3. 连接代码不再存在多个未清理心跳 timeout 叠加关闭同一 socket 的路径。
4. 持续不可达设备的后台重连间隔会逐步增加。
5. 新的 Mac 端 WebSocket 生命周期会写入 `~/.vibedrop/debug.log`。
