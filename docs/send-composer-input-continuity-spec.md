# 发送输入连续性规格

## 背景

VibeDrop 已经能在 Wi-Fi 抖动、WebSocket 断开、智能发现刷新后自动恢复连接。当前剩余体验问题集中在发送页输入框：用户正在用 Android 语音输入时，如果连接自愈触发设备信息同步或发送卡片重渲染，textarea 节点可能被销毁再重建。即使文字草稿能恢复，输入法会话也可能被打断，用户需要重新点语音输入按钮。

## 目标

1. 连接断开、自愈、认证身份补齐、智能发现刷新不应销毁正在编辑的发送 textarea。
2. 正在语音输入、拼音组合输入或普通聚焦输入时，不应程序化重设 textarea 的 `value`、`focus` 或选区。
3. 发送卡片仍要能反映连接状态，状态点、状态文案和按钮可用性可以实时更新。
4. 设备新增、删除、排序、IP/hostname 更新仍要生效，但涉及正在编辑卡片的破坏性 DOM 操作应延后到输入结束。

## 非目标

1. 不改变 Mac 端协议。
2. 不改变连接自愈、心跳、智能发现的触发策略。
3. 不把正在输入的草稿持久化到 localStorage，避免重启后残留敏感内容。

## 技术方案

`renderSendCards()` 从整容器 `innerHTML = ''` 改为按 `deviceId` keyed 复用卡片。已有卡片只更新设备元数据和非输入状态；缺失卡片才创建；不再因为普通连接刷新销毁所有 textarea。

输入框增加轻量输入会话跟踪：`focus`、`blur`、`input`、`compositionstart`、`compositionend` 记录当前设备、最近输入时间和组合输入状态。渲染时如果发现发送 textarea 正在聚焦或处于 composition 状态，则保护该卡片：不移除、不移动、不覆盖输入值、不调用 `focus()` 或 `setSelectionRange()`。

如果渲染期间需要调整卡片顺序或移除已消失设备，但当前有受保护输入框，则先完成新增和非破坏性更新，把排序/删除标记为待处理。输入框 `blur` 或 composition 结束后，如果没有新的活跃输入，再执行一次发送卡片渲染来收敛布局。

## 验证

1. `node --check mobile/src/app.js` 通过。
2. Wi-Fi 抖动导致连接状态变化时，发送页 textarea DOM 节点不被整批重建。
3. 正在语音输入时，绿灯变灰/变绿不应收起键盘或打断语音输入按钮状态。
4. 发送成功仍清空对应输入框和内存草稿；发送失败仍保留草稿。
